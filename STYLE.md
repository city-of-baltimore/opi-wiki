# OPI Wiki Style Guide

How we write on OPI Foundations. The goal is simple: it should read like a smart
colleague explaining the work to another smart colleague. Not marketing, not a
consultant deck, not something a machine generated. Clear, direct, and human.

If you are editing a page, this is the standard. If you are reviewing one, this
is the bar.

## Voice

- **Lead with the point.** The first sentence says what the thing is and what it
  does, in plain words. No throat-clearing, no wind-up.
- **Write like a person talks.** Short, confident sentences. Vary the length,
  but default to short. Read it aloud; if you would not say it, rewrite it.
- **Be concrete.** Name the actual work (schedule meetings, track the budget,
  review the data) instead of the abstraction ("operating backbone").
- **One voice everywhere.** Calm, direct, specific. The About page and a service
  page should sound like the same office wrote them.
- **Active voice, present tense.** "AdminOps tracks the budget," not "The budget
  is tracked by AdminOps."
- **Say it once.** No rule-of-three padding ("focused, measurable, and durable")
  unless each word does real work.

## Jargon: replace these

If a new hire or a resident would not use the word, do not use it. Replace the
left column with the plain-language idea on the right (or just say what happens).

| Instead of | Write |
| --- | --- |
| operating backbone | what it actually does: "keeps the office running," "handles operations" |
| executive performance rhythm / routine / cadence | "regular performance reviews," "CitiStat sessions," "how the city reviews performance" |
| signal-to-solution loop | "how work moves from a problem to a fix" |
| decision-grade (data) | "reliable enough to act on" |
| durable (as filler) | "lasting," or be specific: "so it survives staff turnover" |
| craft (as a noun for the Lab) | "the work," "design and build" |
| north star | "goal," "what we are aiming for" |
| source of truth | "the official record," "the one place to look" |
| at the intersection of | "combines," "works across" |
| high-trust, high-standards | describe it plainly: "we trust each other and hold a high bar" |
| leverage, unlock, empower, robust, seamless, ecosystem, landscape, surface (verb) | plain equivalents, or cut |

This list grows. When you catch a new offender, add it.

## Formatting: prose first

The wiki lost credibility partly because everything was a bullet list or a
table. Default to paragraphs. Use structure only when structure genuinely helps.

- **Bullets** are for a short list of parallel, scannable items (steps, criteria,
  a handful of options). They are not for paragraphs. If a bullet is a full
  sentence or two of explanation, it is prose — write it as prose.
- **No bold-lead bullets as a default.** The `- **Term** — description` pattern
  reads as machine-generated when it is everywhere. Turn it into sentences, or
  into a real definition list only when you are actually defining a set of terms.
- **Tables** are for reference data you look things up in or compare across
  columns: rosters, criteria matrices, KPI definitions, schedules. A one-column
  table, or a two-cell "term / description" table, should be prose or a short
  list.
- **No slogans as blockquotes.** Blockquotes are for real quotations or genuine
  callouts, not inspirational one-liners.
- **Em-dashes: avoid.** Use a period, comma, colon, semicolon, or parentheses.
  A page should have almost none. (Ranges like "T-4 to T-5" are fine.)
- **Headings describe, they don't sloganize.** "How a Stat runs," not "Where the
  magic happens."

## Before and after

**Jargon and abstraction:**

> Before: AdminOps is OPI's internal operations and communications function — the
> operating backbone that keeps the office coordinated, documented, resourced,
> and clearly explained.

> After: AdminOps keeps OPI running. A small office doing citywide work still has
> to coordinate requests, manage resources and records, and explain the work to
> leadership and the City Council. AdminOps does that, so the other teams can
> focus on delivery.

**Bold-lead bullets into prose:**

> Before:
> - **Intake and triage** — one front door for new work, routed to the right team.
> - **Portfolio visibility** — a shared view so leaders can see risk.

> After: New work comes in one front door, where it is sorted and sent to the
> right team. A shared portfolio view keeps that work visible, so leaders can
> see what is at risk and where capacity is tight.

**Keep the table (this one earns it):** a public KPI register with measure,
owner, source, and refresh cadence stays a table. Readers compare those fields.
